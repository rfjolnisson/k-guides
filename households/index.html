What is the Household Model?

Kaptio's recommended approach for managing customer data in the travel industry

The Household Model Explained

A dedicated Account Record Type called Household with all individual travelers (family members, companions) linked as Contacts to a single Household Account record.

Key Distinction: This is a Kaptio-specific approach using a custom Account Record Type. It does NOT use Salesforce's pre-built Household objects from Nonprofit Cloud or Health Cloud.

The Truth: Person Accounts already create Contact records automatically (PersonContactId). Migration extracts those existing Contacts, creates Household Account containers, and reparents the Contacts. Your data is preserved—it's just reorganized.

The Grouping Logic: Who Belongs Together?

This is where migration gets practical. You need algorithmic logic to group Person Accounts into Households. Here are the three common strategies, ranked by reliability:

1

Shared Billing Address (Most Reliable)

Group Person Accounts by identical BillingStreet + BillingPostalCode. This is the gold standard.

WHERE BillingStreet = '123 Main Street'

AND BillingPostalCode = '10001'

GROUP BY (BillingStreet, BillingPostalCode)

Pros: High accuracy, easy to automate, clearly defensible

Cons: Misses households with slightly different address formatting

2

Co-Travelers on Past Bookings (Good)

Analyze Passengers on past Itineraries. If two Person Accounts appear as Passengers on the same booking multiple times, they're a household.

SELECT PersonAccount__c, COUNT(DISTINCT Itinerary__c)

FROM Passenger__c

GROUP BY Itinerary__c HAVING COUNT > 1

Pros: Captures frequent travel companions even with different addresses

Cons: Doesn't work for first-time customers, needs minimum booking threshold

3

Fuzzy Address Matching (Risky)

Use Levenshtein distance or fuzzy matching to group similar addresses. Handles typos and formatting variations.

Requires custom Apex or external tools

High manual review needed for edge cases

Pros: Catches address variations ("Street" vs "St")

Cons: False positives, manual review intensive, complex to implement

Kaptio's Recommendation: Hybrid Approach

  1. Start with exact address matching (90% of households)
  2. Layer co-traveler analysis for remaining records (5-8%)
  3. Manual review edge cases (2-5%)
  4. Create individual households for unmatched records (they can be merged later if needed)

Reality Check: Person Accounts already create Contact records behind the scenes (that's how Salesforce works). Those Contacts have all the data you need. Migration is really just: (1) Create new Household Accounts, (2) Reparent existing Contacts to those Households, (3) Update lookups on Passengers/Itineraries/Trips. The Contact data doesn't move—just the relationships.

The Hard Part: Data Quality

Migration difficulty isn't technical complexity—it's data quality. If your addresses are messy ("123 Main St" vs "123 Main Street"), grouping logic fails. The actual data transformation is straightforward. The work is in data cleansing and validation.

Before Migration: Clean Your Data
  • Standardize address formatting
  • Merge duplicate Person Accounts
  • Fill in missing postal codes
  • Run deduplication reports

Migration Script Logic (Simplified)

// Step 1: Group Person Accounts by address
Map> householdGroups = groupByAddress(personAccounts);

// Step 2: Create new Household Accounts
for (String addressKey : householdGroups.keySet()) {
    List personAccounts = householdGroups.get(addressKey);
    
    // Create Household Account
    Account household = new Account(
        Name = generateHouseholdName(personAccounts),  // e.g., "Smith Household"
        RecordTypeId = householdRecordTypeId,
        BillingStreet = personAccounts[0].BillingStreet,
        BillingPostalCode = personAccounts[0].BillingPostalCode
    );
    insert household;
    
    // Step 3: Reparent Contacts to new Household
    List contactsToUpdate = [
        SELECT Id, AccountId 
        FROM Contact 
        WHERE AccountId IN :getPersonAccountIds(personAccounts)
    ];
    
    for (Contact c : contactsToUpdate) {
        c.AccountId = household.Id;  // Reparent to Household
    }
    update contactsToUpdate;
    
    // Step 4: Update child records (Passengers, Itineraries, Trips)
    // ... update lookups to point to new Household Account ...
}

// Step 5: Delete old Person Accounts (data is safe in Contacts)
delete oldPersonAccounts;
